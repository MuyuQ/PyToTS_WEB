## 🎯 为什么这个很重要

你在 Python 代码里写了漂亮的类型注解：

```python
def process_user(user: dict) -> str:
    return user["name"]  # 类型检查器不会报错

# 但运行时...
process_user({"age": 25})  # KeyError: 'name'
```

类型注解写了，运行时却报错。Python 的类型注解像"建议"而非"规则"。

TypeScript 不一样：

```typescript
function processUser(user: { name: string }): string {
  return user.name;
}

// 编译时就报错，根本跑不起来
processUser({ age: 25 }); // Error: Property 'name' is missing
```

这就是"运行时类型"和"编译时类型"的区别，也是 Python 开发者转向 TypeScript 最大的思维转变。

---
## 💡 核心概念

### 类型注解 = 盒子上的标签 vs 带锁的盒子

| 类型系统        | 类比             | 特点                                     |
| --------------- | ---------------- | ---------------------------------------- |
| Python 类型注解 | 贴在盒子上的标签 | 告诉你"应该"放什么，但放错了也不会阻止你 |
| TypeScript 类型 | 带锁的盒子       | 放错东西直接打不开，编译时就报错         |

### 类型推断 = 智能管家

```python
# Python: 需要手动标注
name: str = "Alice"
```

```typescript
// TypeScript: 智能管家自动推断
const name = "Alice"; // TypeScript 知道这是 string 类型

// 不用写类型注解，TypeScript 也能猜出来
const numbers = [1, 2, 3]; // number[]
const user = { name: "Bob", age: 25 }; // { name: string; age: number }
```

TypeScript 的类型推断像智能管家，看一眼就知道你要放什么。

### 联合类型 = 多功能插座

```python
# Python: Union 类型
from typing import Union

def process(value: Union[int, str]) -> str:
    return str(value)
```

```typescript
// TypeScript: 联合类型（更简洁的语法）
function process(value: number | string): string {
  return String(value);
}
```

联合类型像多功能插座，可以插两种或多种插头。

### 泛型 = 可调节的架子

```python
# Python 泛型
from typing import TypeVar, Generic, List

T = TypeVar('T')

class Box(Generic[T]):
    def __init__(self, value: T):
        self.value = value

    def get(self) -> T:
        return self.value
```

```typescript
// TypeScript 泛型（更简洁）
class Box<T> {
  constructor(private value: T) {}

  get(): T {
    return this.value;
  }
}

// 使用
const numberBox = new Box(42); // Box<number>
const stringBox = new Box("hello"); // Box<string>
```

泛型像可调节的架子，可以放不同尺寸的书。

## 📚 Python 回顾

```python
# Python 类型注解
def greet(name: str) -> str:
    return f"Hello, {name}"

# 可选类型
from typing import Optional

def find_user(id: int) -> Optional[str]:
    if id > 0:
        return "user"
    return None

# 联合类型
from typing import Union

def process(value: Union[int, str]) -> str:
    return str(value)

# 类型别名
from typing import List, Dict

UserDict = Dict[str, int]
UserList = List[UserDict]
```

---
## 🔧 TypeScript 等价写法

```typescript
// TypeScript 类型注解
function greet(name: string): string {
  return `Hello, ${name}`;
}

// 可选类型 (使用 null 或 undefined)
function findUser(id: number): string | null {
  if (id > 0) {
    return "user";
  }
  return null;
}

// 联合类型（语法更简洁）
function process(value: number | string): string {
  return String(value);
}

// 类型别名
type UserDict = Record<string, number>;
type UserList = UserDict[];
```

## ⚠️ 差异与常见陷阱

### 1. Optional vs T | null

```python
# Python: Optional[T] 等价于 T | None
from typing import Optional

def find_user(id: int) -> Optional[str]:
    return None
```

```typescript
// TypeScript: 直接用联合类型
function findUser(id: number): string | null {
  return null;
}

// 或者 undefined（更常见）
function findUser2(id: number): string | undefined {
  if (id > 0) return "user";
  // 不写 return，默认返回 undefined
}
```

> **注意**: TypeScript 的 `null` 和 `undefined` 是两个不同的值。`null` 表示"有值，但是空的"，`undefined` 表示"没有值"。

### 2. 类型位置

| Python            | TypeScript             | 说明           |
| ----------------- | ---------------------- | -------------- |
| `name: str`       | `name: string`         | 变量声明       |
| `def f() -> int:` | `function f(): number` | 返回值位置不同 |
| `Optional[str]`   | `string \| null`       | 可选类型语法   |
| `Union[int, str]` | `number \| string`     | 联合类型语法   |

### 3. 运行时检查 vs 编译时检查

```python
# Python: 类型注解运行时不强制
def add(a: int, b: int) -> int:
    return a + b

add("hello", "world")  # 运行时正常，返回 "helloworld"
```

```typescript
// TypeScript: 编译时检查
function add(a: number, b: number): number {
  return a + b;
}

add("hello", "world"); // 编译错误！不能把 string 传给 number
```

## 💬 开发者对话

🐍 Python 开发者："TypeScript 的类型检查太严格了，我写个简单函数都要报错！"

🦀 TS 开发者："严格才能提前发现问题啊。你想想，Python 的类型注解写错了，运行时才发现。TypeScript 编译时就告诉你错了。"

🐍 Python 开发者："但有时候我就是想传个 `any` 类型的值，Python 多灵活..."

🦀 TS 开发者："TypeScript 也有 `any`，但建议用 `unknown`。`any` 是关掉检查，`unknown` 是延迟检查，更安全。"

```typescript
// ❌ any: 关掉所有检查（危险）
let data: any = fetchUser();
data.foo.bar.baz; // 编译通过，运行时可能爆炸

// ✅ unknown: 延迟检查（安全）
let data: unknown = fetchUser();
if (typeof data === "object" && data !== null) {
  // 类型收窄后才能使用
}
```

🐍 Python 开发者："那 `Optional[T]` 和 `T | null` 有什么区别？"

🦀 TS 开发者："本质一样，都是表示'可能有值，可能没有'。但 TypeScript 更灵活："

```typescript
// 这些都是合法的"可选"类型
type MaybeString = string | null;
type MaybeNumber = number | undefined;
type MaybeUser = User | null | undefined;

// Python 需要这样写
# MaybeString = Optional[str]  # 只能是 str | None
```

🐍 Python 开发者："泛型看起来差不多..."

🦀 TS 开发者："是的！但 TypeScript 的泛型推断更智能："

```typescript
// 不用写类型参数，TypeScript 自动推断
function identity<T>(value: T): T {
  return value;
}

const str = identity("hello"); // 自动推断 T = string
const num = identity(42); // 自动推断 T = number
```

## 🤔 试试看

### 预测输出

下面这段 Python 代码，类型注解和实际传入的值不一致，但能运行。猜猜结果是什么？

```python
from typing import List

def sum_numbers(numbers: List[int]) -> int:
    return sum(numbers)

# 类型注解说是 List[int]，但传入的是 List[str]
result = sum_numbers(["1", "2", "3"])
print(result)  # 输出什么？
```

<details>
<summary>点击查看答案</summary>

```
TypeError: unsupported operand type(s) for +: 'int' and 'str'
```

Python 的类型注解只是"建议"，运行时不会检查。`sum(["1", "2", "3"])` 尝试把字符串相加，`"1" + "2"` 得到 `"12"`，然后 `"12" + "3"` 得到 `"123"`...

等等，Python 的 `sum()` 默认起始值是 `0`（整数），所以是 `0 + "1"`，类型错误！

这正是 TypeScript 要在编译时阻止的问题。

</details>

### 找错误

下面这段 TypeScript 代码有类型错误，你能找到吗？

```typescript
interface User {
  name: string;
  age: number;
}

function greetUser(user: User): string {
  return `Hello, ${user.name}`;
}

// 调用
const result = greetUser({
  name: "Alice",
  age: "25", // 有问题吗？
});
```

<details>
<summary>点击查看答案</summary>

错误在 `age: "25"`。

`age` 应该是 `number` 类型，但 `"25"` 是字符串。

TypeScript 编译时报错：

```
Type 'string' is not assignable to type 'number'.
```

修复方法：`age: 25`（数字，不是字符串）

</details>

### 代码翻译

将以下 Python 代码转换为 TypeScript：

```python
from typing import Optional, List

def find_first_user(users: List[str]) -> Optional[str]:
    if users:
        return users[0]
    return None
```

<details>
<summary>点击查看答案</summary>

```typescript
function findFirstUser(users: string[]): string | undefined {
  if (users.length > 0) {
    return users[0];
  }
  return undefined;
}
```

要点：`List[str]` 转成 `string[]`，`Optional[str]` 转成 `string | undefined`（TypeScript 更常用 undefined 表示"没有"），`if users:` 转成 `if (users.length > 0):`。

</details>

## 📊 优势对比

| 特性       | Python        | TypeScript       |
| ---------- | ------------- | ---------------- |
| 类型推断   | 有限          | 强大             |
| 编译检查   | 无（需 mypy） | 有               |
| 泛型支持   | 有            | 有（语法更简洁） |
| 接口       | Protocol      | interface        |
| 联合类型   | Union[A, B]   | A \| B           |
| 运行时类型 | 保留          | 擦除             |

## 🎓 面试实战

### 基础问题

**Q: TypeScript 的 `any` 和 `unknown` 有什么区别？**

:::tip[答案要点]

- `any`: 关闭类型检查，可以访问任意属性，危险
- `unknown`: 类型安全的 any，必须先类型收窄才能使用

```typescript
let anyValue: any = "hello";
anyValue.foo(); // 编译通过，运行时崩溃

let unknownValue: unknown = "hello";
unknownValue.foo(); // 编译错误！
if (typeof unknownValue === "string") {
  unknownValue.toUpperCase(); // OK
}
```

:::

**Q: 什么时候使用 `interface` vs `type`？**

:::tip[答案要点]

- `interface`: 定义对象形状，可以被 extends/implements，声明合并
- `type`: 类型别名，支持联合类型、交叉类型、映射类型

```typescript
// interface 可以被扩展
interface User {
  name: string;
}
interface User {
  age: number; // 声明合并
}

// type 支持更多类型操作
type ID = string | number;
type UserWithID = User & { id: ID };
```

经验法则：优先用 `interface`，需要高级类型操作时用 `type`。
:::

### 进阶问题

**Q: TypeScript 如何处理 `null` 和 `undefined`？**

:::tip[答案要点]
TypeScript 有 `strictNullChecks` 选项：

```typescript
// strictNullChecks: true（推荐）
let name: string = null; // 错误！
let name: string | null; // OK

// 可选属性自动加上 | undefined
interface User {
  name: string;
  age?: number; // number | undefined
}
```

:::
